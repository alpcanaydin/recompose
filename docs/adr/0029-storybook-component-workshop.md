# 0029: Storybook component workshop, blocking gates, and the assistant posture

**Status**: Accepted
**Date**: 2026-07-24

## Context

The eighth infrastructure-queue item needed a component workshop. The renderer held six components, styled through the two-tier Tailwind token system. Architecture Decision Record (ADR) 0009 already named Storybook as that system's expected second consumer. Feature-Sliced Design (FSD) boundaries stay Steiger-enforced. Component tests already ran in a real Chromium browser through Vitest, against a hand-rolled fake of the `window.recompose` bridge. Chromatic visual regression waits as queue item ten and builds on this job. The maintainer locked three choices in a brainstorm before implementation started. The scope covers infrastructure, seed stories, and a story-required policy. The gate blocks on story tests and on accessibility. Model Context Protocol (MCP) wiring stays project scoped.

## Decision

- **Storybook 10.5 lives inside `apps/desktop`** on the `@storybook/react-vite` framework, authored through Component Story Format (CSF) factories (`preview.meta({ component })`, then `meta.story({ args })`). See Alternatives for why a root workspace and a `packages/ui` extraction both lost.
- **`.storybook/main.ts` replicates the renderer's Vite pieces, because Storybook never reads `electron.vite.config.ts`.** Its `viteFinal` hook re-declares the `@renderer` alias and adds the Tailwind Vite plugin by hand. The alias merges through Vite's own `mergeAlias` helper rather than a raw object spread, because `resolve.alias` can carry either an object or an array under `AliasOptions`, and a plain spread of an array-shaped value trips the type-aware `typescript/no-misused-spread` oxlint rule.
- **Prop extraction pins `reactDocgen: 'react-docgen'`, not the TypeScript-aware `react-docgen-typescript` the original design named.** The TypeScript-aware extractor hard-crashes against this repository's `typescript@7.0.2` pin: the native compiler's root export no longer exposes `ts.sys`, a value that extractor's static analysis depends on. Storybook's own documented default filled the gap. Until an extractor release supports TypeScript 7, hand-written JSDoc on props carries more of the manifest's documentation weight than upstream guidance assumes.
- **Story tests join `pnpm test` as a third Vitest project.** `@storybook/addon-vitest`'s `storybookTest` plugin reuses the existing Playwright Chromium provider already wired for browser tests, so every story runs as a real browser test alongside the `unit` and `browser` projects. Accessibility runs inside that same test through `parameters.a11y.test: 'error'`, so a violation fails the story outright instead of just logging a warning. A `storybook build` smoke step joins the required `check` job in Continuous Integration (CI), catching a broken configuration or story even when every test passes. Lefthook's pre-commit chain runs no test job at all today (it stops at typecheck), so CI stays the natural home for a gate this heavy.
- **The pull-request meta-gate gained a story-required rule.** A diff that adds a new component file under a renderer `ui/` segment without a sibling `*.stories.tsx` fails the `meta` job. The escape is the `stories-exempt` label plus a `Stories-exempt: <reason>` line in the pull request (PR) body, and the weekly ruleset-bypass audit now lists `stories-exempt` alongside the existing `tdd-exempt` and `adr-exempt` labels.
- **Component, prop, and story JSDoc is the one sanctioned exception to the no-comments rule.** Storybook's manifest generation reads that JSDoc into `/manifests/components.json` and its human-readable counterpart, so a coding assistant working against this codebase can browse a curated catalogue instead of re-reading every component's source.
- **One scoped `.oxlintrc.json` override covers `.storybook/recompose-bridge.tsx`.** The decorator narrows an unknown story parameter with `as BridgeParameters`, a sanctioned cast this file needs to install the typed bridge fake. The override turns off `typescript/consistent-type-assertions` and `typescript/no-unsafe-type-assertion` for that one path, mirroring the existing precedent already carved out for `apps/desktop/src/preload/index.ts`.
- **The blocking accessibility gate caught a real violation on its first day.** The providers page heading failed a color-contrast check under `parameters.a11y.test: 'error'`, and the fix moved `apps/desktop/src/renderer/src/pages/providers/ui/providers-page.tsx` from `text-ink-secondary` to `text-ink`. The gate worked exactly as designed: it blocked a shipped contrast bug before any story-only landing could hide it.
- **`EmptyState` moved into a `pages/home` slice.** The index route at `apps/desktop/src/renderer/src/app/routes/index.tsx` now delegates to that slice's public application programming interface (API) instead of owning the empty-state markup inline, which lines the route up with the thin-adapter rule ADR-0017 already set for every other route.
- **The theme toolbar flips `color-scheme` classes, not inline styles.** `@storybook/addon-themes`'s `withThemeByClassName` decorator swaps `scheme-light` and `scheme-dark` on the preview's `html` parent element, because the token tiers already resolve through the Cascading Style Sheets (CSS) `light-dark()` function and only need the ambient `color-scheme` to pick a branch.
- **MCP wiring stays project scoped and development only.** `@storybook/addon-mcp` lives in `main.ts`'s `addons` array, so the Model Context Protocol server exists only while `storybook dev` runs, never in the packaged app and never in the static `storybook build` output. A committed `.mcp.json` entry points a client assistant at `http://localhost:6006/mcp`. The official Storybook Claude Code plugin installs at project scope through `.claude/settings.json`'s `enabledPlugins`, sourced from the `storybookjs/mcp@main` marketplace entry in `extraKnownMarketplaces`. Both the addon and its generated manifests are experimental and development-time only, so no packaged-product risk follows from either.
- **Implementation plans under `docs/superpowers/plans/` left both prose gates.** The change rode this job rather than waiting for a dedicated one: the Vale glob (in both the root `lint:prose` script and the CI `vale_flags` input) and the cspell `ignorePaths` list now exclude that directory, because a plan is an internal execution artifact, not shipped documentation.

## Alternatives

- **A root-level Storybook workspace**: rejected. Every story today lives entirely inside `apps/desktop`, the sole consumer. Routing Storybook through a root workspace package would add an indirection layer with nothing on the other side of it to justify the boundary crossing.
- **Extracting a `packages/ui` package ahead of Storybook**: rejected on the same FSD and You Aren't Gonna Need It (YAGNI) grounds ADR-0017 already applied to routing. FSD demands proven multi-use before a shared package earns its boundary, and today's six components are page-local. Storybook documents them in place. Extraction waits for a real second consumer.
- **`react-docgen-typescript` for prop extraction**: the original design's pick, and still the stronger extractor in principle. Rejected in practice once it crashed against `typescript@7.0.2`'s missing `ts.sys` export. Revisiting this choice is a queued follow-up once the extractor catches up to TypeScript 7.
- **A raw object spread to merge the Vite alias**: works until `resolve.alias` arrives as an array, which `AliasOptions` allows. `mergeAlias` is the type-safe merge Vite ships for exactly this shape, so the hook uses it instead of re-deriving the same logic by hand.
- **A lefthook job for story tests**: rejected. Pre-commit today stops at typecheck and runs no test job of any kind, and the full Vitest suite, story tests included, spins up a real Playwright Chromium instance that takes too long for a commit-time hook's budget. The required CI `check` job is the gate's honest home.
- **A Mock Service Worker layer inside stories**: unnecessary. The renderer never talks to raw Hypertext Transfer Protocol. It talks through the typed `window.recompose` bridge, and the existing fake bridge decorator from the browser tests covers every story scenario directly.

## Consequences

**Good**: every renderer component (styled or wired) has a runnable, visually inspectable story, and the story-required gate keeps that property from eroding as new components land. The manifest JSDoc gives a coding assistant a curated, static-analysis-backed catalogue instead of a raw source read. The blocking accessibility check already caught and fixed one real contrast bug before merge. The assistant posture keeps every piece of that automation, the addon, the manifests, and the plugin, out of the packaged product entirely. The theme toggle exercises both color schemes through the same CSS the shipped app uses, not a parallel test-only theme system.

**Bad, and accepted**: prop-table quality depends on hand-written JSDoc until `react-docgen` gains TypeScript 7 support, a weaker guarantee than the TypeScript-aware extractor the original design assumed. The oxlint sanctioned-cast override now covers two files instead of one. A third file needing the same escape should raise the question of whether the cast belongs in a shared boundary helper instead. Implementation plans under `docs/superpowers/plans/` no longer get a prose or spelling check, so a typo there survives, a deliberate trade against the gate's real cost against non-shipped documents.
