# ADR-0015: Repo Hardening Layer

**Status**: Accepted
**Date**: 2026-07-23

## Context

recompose is a public, single-maintainer GitHub repository. The gates from ADR-0006/0007/0011/0013 cover local and CI quality, secrets, the forbidden alias, and review completeness, but eight gaps remained: merges could land against a stale base, branch names were unconstrained, secret scanning/push protection were off, dependency freshness was unbounded, CI runners had no egress visibility, the coverage floor could only move by hand, unused dependencies/exports had no gate, and ruleset bypasses (the admin escape hatch every ruleset in this repo grants) were invisible. All eight are addressed together as one hardening pass.

## Decision

1. **Stale-base merges close**: `.github/rulesets/main.json` sets `required_status_checks.strict_required_status_checks_policy: true`, applied live to ruleset id `19530816` via `gh api --method PUT`. A PR's head must now be up to date with `main` before its checks count.

2. **Branch name pattern**: the spec'd `branch_name_pattern` rule (regex metadata restriction) returned `422 Invalid rule 'branch_name_pattern'` — that rule type requires GitHub Enterprise Cloud, not available on GitHub Free. Substituted the standard Free-tier equivalent: ruleset `branch-names` (id `19609433`) with a `creation` rule and `conditions.ref_name.exclude: ["refs/heads/main", "refs/heads/worktree-*", "refs/heads/renovate/**"]` — creation of any branch not matching an allowed prefix is blocked. Same practical effect (worktree-/renovate/ prefixes only), reached through glob excludes instead of a regex rule.

3. **Secret scanning + push protection**: both set to `enabled` via `gh api --method PATCH repos/alpcanaydin/recompose -f security_and_analysis[secret_scanning][status]=enabled -f security_and_analysis[secret_scanning_push_protection][status]=enabled`. Free on public repos.

4. **pnpm `minimumReleaseAge: 4320`** (3 days, matching Renovate's stability window) added to `pnpm-workspace.yaml`. This broke plain `pnpm install`/`pnpm exec` immediately: two already-locked entries (`@typescript-eslint/types@8.65.0`, `baseline-browser-mapping@2.10.44`) were published inside the new 3-day window, predating the policy. Resolution: CI's `pnpm install --frozen-lockfile` steps gained `--trust-lockfile`, pnpm's documented flag for exactly this case ("CI runs against an already-verified lockfile") — CI reproduces a fixed, already-reviewed lockfile and does not need to re-litigate its supply-chain freshness on every run. Local/interactive installs and `pnpm add` still enforce the full policy, which is the point of the ratchet — but any subsequent edit to `pnpm-workspace.yaml` (a new `overrides` entry, another dependency add) forces pnpm to re-resolve locally and re-hits these same two entries, since `--trust-lockfile` only covers CI's frozen install step, not a real resolution pass. Both were therefore added to `minimumReleaseAgeExclude` alongside knip once the item 7 override needed a clean local/CI resolution to go through; they can come back out once they naturally clear the 3-day window. Adding knip (item 7) required the same friction once more; pnpm auto-recorded the override as `minimumReleaseAgeExclude: [knip@6.29.0]` in `pnpm-workspace.yaml`, which is the intended per-package escape hatch for a deliberately fresh add.

5. **harden-runner in audit mode**: `step-security/harden-runner@bf7454d06d71f1098171f2acdf0cd4708d7b5920 # v2.20.0` added as the first step of all eight `ubuntu-latest` jobs in `ci.yml` (`changes`, `check`, `gitleaks`, `audit`, `zizmor`, `actionlint`, `commitlint-pr`, `ci-success`), `egress-policy: audit`. Audit, not block, is deliberate — block mode needs an accurate egress allowlist first, and a blind one breaks CI outright. Plan: let audit logs accumulate for a few weeks of real CI runs, derive the allowlist from what each job actually contacts, then flip every job to `egress-policy: block` with that allowlist in a follow-up ADR.

6. **Coverage ratchet — reverted**. `autoUpdate: true` added to the shared `thresholds` block in `vitest.shared.ts` made `pnpm --filter @recompose/desktop run test` fail outright: `Unable to parse thresholds from configuration file: Expected config.test.coverage.thresholds to be an object`. Vitest 4.1.10's `autoUpdate` rewrites the thresholds object via source-level AST edits to the project's own vitest config; it cannot locate a literal object when thresholds arrive via `...coverageDefaults` spread from a shared, imported module. That indirection is exactly ADR-0012's single-source-of-thresholds design, so the ratchet is incompatible with the current architecture, not a bug to work around. Reverted in full; thresholds stay hand-edited in `vitest.shared.ts`.

7. **knip**: `knip@6.29.0` pinned exact (`-E`) as a root devDependency. Root `knip.json` is nearly empty (`{"$schema": ...}`) — knip's electron-vite and vitest plugins auto-detect all entry points (`src/main/index.ts`, `src/preload/index.ts`, `src/renderer/src/app/main.tsx`, both vitest configs) from `electron.vite.config.ts` and `vitest.config.ts` with zero configuration; an explicit `entry` list only produced "redundant entry pattern" config hints. One real finding surfaced: `fast-check` in `apps/desktop/package.json` has no direct import anywhere — it is consumed only through `@fast-check/vitest`, which already declares `fast-check` as its own dependency. Read as a deliberate, explicit pin (controlling the property-testing engine's version independent of `@fast-check/vitest`'s `^3.0.0 || ^4.0.0` range) rather than dead weight, so it was not deleted; instead `knip.json` carries a narrow `workspaces["apps/desktop"].ignoreDependencies: ["fast-check"]`. That alone only kept the pin from being flagged as dead — nothing guaranteed `@fast-check/vitest`'s own `fast-check` dependency actually resolved to it beyond lockfile dedupe, so `pnpm-workspace.yaml` also gained an `overrides: { fast-check: "4.9.0" }` entry, making the version genuinely single-sourced rather than assumed. Wired up: root script `lint:dead: knip`, lefthook pre-commit job `dead` (priority 1), and a standalone `pnpm run lint:dead` CI step in the `check` job — standalone because `lint:boundaries`/`lint:fsd` (referenced in the original spec) don't exist on this branch; they land via unmerged PR #34 and should be combined with `lint:dead` into one line once that PR merges.

8. **Bypass visibility**: `.github/workflows/ruleset-bypass-audit.yml` — weekly (`0 6 * * 1`) plus `workflow_dispatch`, `permissions: {issues: write}`, one `ubuntu-latest` job (harden-runner audit step first). Queries `repos/{repo}/rulesets/rule-suites?rule_suite_result=bypass&time_period=week`. That endpoint requires the repo's Administration:read permission, which the Actions `GITHUB_TOKEN` permissions block cannot grant — there is no `administration:` key — so this step authenticates with `secrets.RULESET_AUDIT_TOKEN`, a fine-grained PAT with Administration:read + Metadata:read on this repo, created by the repo owner and stored as a repo secret. A guard step runs first and fails the job with `RULESET_AUDIT_TOKEN secret missing (fine-grained PAT, Administration:read)` if the secret is empty: the workflow stays red until the secret is configured, deliberately — a silently dead audit is worse than a red run. When bypasses are found, the workflow upserts a single stable-titled issue ("Ruleset bypass audit") instead of creating a new one every week: it looks up an existing open issue with that title via `gh issue list --search` (using the default `GITHUB_TOKEN`, which already carries `issues: write`), comments the week's findings onto it if found, and creates the issue only if none exists yet. Verified live: the endpoint already returns 3 entries this week. Rule suites record PUSH evaluations, not admin API calls, so these are not the `gh api PUT/POST` calls from items 1–2 — they are three admin-bypass merges to `main`, made with `--admin` while the CodeRabbit review quota was exhausted.

## Alternatives

- **Merge queue instead of strict required status checks**: gives the same no-stale-base guarantee but adds queue latency for a single-maintainer repo with low PR volume; strict checks give the core benefit (checks run against the true merge result) at zero added latency.
- **Signed commits**: rejected as theater for a single-maintainer repo — the maintainer is the only signer and the only one who could forge a commit as themselves; no attacker model this addresses that push protection and branch rules do not already cover.

## Consequences

**Good**: no merge can land against a stale base; branch creation outside `worktree-`/`renovate/` is blocked at the platform; secrets are scanned and blocked before they leave the machine on the GitHub side too, not just locally; the lockfile can no longer silently pick up a package published in the last three days; every CI job now logs its network egress for later allowlisting; dead dependencies/exports have a real gate (currently green); ruleset bypasses — the one hole every bypass_actor entry deliberately leaves open — get a weekly paper trail instead of none.

**Bad**: every PR now needs an "update branch" (or fresh push) if `main` moves underneath it before merge — friction on every rebase-needed PR; the branch-name ruleset blocks off-pattern branch creation for everyone except the admin actor, whose `bypass_mode: always` means a legitimate one-off branch name (e.g. a hotfix style outside the two allowed prefixes) is created silently with no block and no prompt — it only surfaces after the fact, in the weekly bypass audit; harden-runner in audit mode observes egress but blocks nothing yet — a compromised action in a dependency could still exfiltrate until the block-mode follow-up lands; `minimumReleaseAge` needs the `--trust-lockfile`/`minimumReleaseAgeExclude` escape hatches understood by whoever next touches CI or adds a dependency, or they will look like inexplicable exceptions to the policy; the weekly report will list every admin-bypass merge to `main`, not just the ones worth a second look — that noise is the point of the audit (bypass visibility) and is accepted deliberately, but if volume ever drowns signal, per-rule filtering down to approval-only bypasses is the follow-up.
