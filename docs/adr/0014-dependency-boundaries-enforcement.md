# 0014: Dependency boundaries enforced by dependency-cruiser and Steiger

**Status**: Accepted
**Date**: 2026-07-23

## Context

Architecture Decision Record (ADR) 0010 chose the tools. The folder-structure spec wrote the dependency rules: Electron process isolation, the engine/contracts/desktop package directions, and no circular imports. It also required Feature-Sliced Design (FSD) layer discipline in the renderer. Those rules existed only as prose ("target state"). ADR-0011's lesson applies: prose rules drift, but machine-checked rules don't.

## Decision

- **One whole-repo import graph at the root.** `.dependency-cruiser.cjs` and `steiger.config.ts` live at the repo root; `lint:boundaries` (`depcruise apps`) and `lint:fsd` (`steiger â€¦ --fail-on-warnings`) are root scripts in the same gate layer as `fmt:check`, not turbo tasks, because cross-package rules need the full graph, and both tools finish in under a second.
- **Eight dependency-cruiser rules, all `error`**: `no-circular`, three Electron process-isolation rules (renderer/main/preload can't import across process lines; the preload type declarations are the one exception), and four pre-staged package rules (`engine-no-electron`, `engine-only-contracts`, `desktop-not-into-engine`, `headless-scope`) targeting paths that don't exist yet, so they take effect the day a package opens. The single follow-up when `packages/` opens: add `packages` to the scan arguments (`depcruise apps packages`); depcruise errors on nonexistent paths, and this decision rejects `includeOnly`, because it filters node_modules edges out of the graph, which disables `engine-no-electron` without warning.
- **Steiger with `fsd.configs.recommended` and `--fail-on-warnings`** over `apps/desktop/src/renderer/src`: warnings fail too; a rule disable requires a config entry, which review sees.
- **Stages: pre-commit + CI** (ADR-0006/0007 layering): two lefthook jobs plus one CI `check`-job step running the same scripts.
- **`@swc/core` (pinned 1.15.46) is a root devDependency** solely because dependency-cruiser 18.1.0 hard-rejects typescript >= 7 (the repo pins 7.0.2). Without swc present, the scan covered zero `.ts`/`.tsx` files, and the empty gate gave no warning. With swc installed as an automatically detected fallback, all TS files get scanned and type-only imports become visible (proven empirically). The trap: never enable dependency-cruiser's explicit `parser: 'swc'` option, because its swc integration lacks tsx support and crashes on any JSX file. `tsPreCompilationDeps: true` is currently inert for the same TS-7 reason, and stays in the config without causing harm. A future "config cleanup" reversing either decision re-opens the hole.
- **`@steiger/toolkit` (pinned 0.2.3, matching upstream's own pin) is a root devDependency** because steiger/plugin published type declarations reference it without declaring the dependency; without it, oxlint's type-aware rules fail on `steiger.config.ts` (tsc stays silent only via skipLibCheck). This pin carries a benign unmet-peer warning concerning vitest 4, known and accepted.
- **Boundary gate extended to max strictness with six more `error` rules**: `no-phantom-deps` (imports of npm packages undeclared in the owning `package.json`), `not-to-unresolvable` (imports that can't resolve on disk, with a single sanctioned exception for the electron-vite `?asset` query import) paired with an `enhancedResolveOptions` resolver fix, `no-orphans` (dead modules, with sanctioned exceptions for `.d.ts` files, test files, and process entry points), `not-to-test` (production code banned from importing `*.test.ts(x)`/`*.browser.test.tsx` files), `no-deprecated-core` (deprecated Node.js core modules: `punycode`, `domain`, `constants`, `sys`, `_linklist`, `_stream_wrap`), and `no-duplicate-dep-types` (an import edge classified under more than one non-type-only dependency type, for example listed in both `dependencies` and `devDependencies`).
- **The resolver fix needed two options, not one.** `conditionNames: ['import', 'require', 'node', 'default', 'types']` alone left `@tailwindcss/vite`, `@vitejs/plugin-react`, and `@fast-check/vitest` unresolvable, because dependency-cruiser's own `DEFAULT_RESOLVE_OPTIONS` hardcodes `exportsFields: []` for backwards compatibility with enhanced-resolve 4, which disables the package.json `exports` field entirely regardless of `conditionNames`. Packages that publish only an `exports` map (no legacy `main`) stayed unresolvable until the config added `exportsFields: ['exports']` alongside `conditionNames`. Proven empirically: a side-by-side config diff showed `couldNotResolve: true â†’ false` for all three externals only once the config set both options.
- **This decision adds `options.exclude: { path: '(^|/)(out|coverage)/' }`**, discovered while proving `no-orphans`: `depcruise apps` was scanning `apps/desktop/out/` (electron-vite build output) and `apps/desktop/coverage/` (vitest coverage HTML report), both gitignored and neither part of the source tree, because no prior rule had an unconstrained `to: {}` that would touch them. `no-orphans`'s `from: { orphan: true }, to: {}` was the first rule to reach every module, exposing a pre-existing scan-scope gap: the gate's cleanliness depended on whether a developer had `out/`/`coverage/` on disk locally, not on repo state. Excluding both directories makes the scan deterministic and dropped the module count from 37 to 31, matching this task's own stated starting baseline and confirming the exclude reflects the graph the gate was always meant to see.
- **`not-to-dev-dep` rejected, not deferred.** In an Electron app the entire toolchain (build plugins, test runners, and `electron` itself) is a devDependency by design; the rule would drown in exceptions instead of catching anything real.
- **`missing-typescript-transpiler` is a tolerated, cosmetic warning** on every `lint:boundaries` run. dependency-cruiser probes for a typescript package in the `>=2.0.0 <7.0.0` range; the repo pins `typescript@7.0.2`, which is outside that range, so the probe always reports it as "missing" even though `typescript` is present. `@swc/core` (mentioned earlier) does the actual scanning, so the warning has no effect on what gets cruised. The warning stays on stderr instead of getting suppressed, because piping/filtering stderr risks swallowing a real dependency-cruiser error alongside it. It disappears on its own once dependency-cruiser ships support for typescript >= 7.
- **Three transitive counterparts added for the Electron process-isolation rules**: `renderer-isolated-transitive`, `main-not-into-renderer-transitive`, `preload-isolated-transitive`, each the direct rule's `from`/`to` with `to.reachable: true` added. The direct rules only ban a single edge crossing a process boundary; a file outside `main`/`preload`/`renderer` (for example, a future `src/common/`) could bridge renderer code to `main` code without ever appearing as a `from` or `to` in the direct rules, since neither pattern matches the bridging file. Proven real: `apps/desktop/src/renderer/src/indirect-probe.ts` importing `apps/desktop/src/bridge-probe.ts` (outside all three process folders) importing `apps/desktop/src/main/windows/window-options.ts` fired only `renderer-isolated-transitive`; `renderer-isolated` stayed silent on the rendererâ†’bridge edge, confirming the gap existed before this rule closed it. dependency-cruiser 18.1.0 evaluates `to.reachable: true` as a module-level ("moduleOnlyRule") check, computed by a dedicated reachability-analysis pass (`src/analyze/derive/reachable.mjs`) rather than the per-edge matcher the direct rules use. This is why the config keeps both a direct and a transitive rule: the direct rule's violation message names the exact one-hop edge, and the transitive rule's message includes the full path through the intermediary.

## Alternatives

- **Per-package turbo tasks**: fits turbo's caching model but splits the one graph cross-package rules need; sub-second tools gain nothing from caching.
- **eslint-plugin-boundaries**: needs an ESLint toolchain, which this repo skips in favor of the oxlint ecosystem (ADR-0010).
- **CI-only**: catches violations after they're public; the local hook stops them at commit time, CI remains the net for `--no-verify`.

## Consequences

**Good**: every dependency rule from the folder-structure spec is now an exit code. No violation can reach a local commit or a remote merge. The engine's purity rules exist before the engine does, so its first line of code is already fenced.

**Bad**: pre-commit gains two more seconds. The scan-argument change when `packages/` opens is a manual step, and only this decision record's text will remind anyone to make it. Steiger can't take custom rules yet, so the FSD recommended set caps renderer rules. The electron rule matches specifiers by name (`^electron(/|$)`, including subpaths like `electron/utility`), because pnpm's isolated layout keeps them unresolvable from unopened packages. The swc and steiger/toolkit pins are upgrade-coupled to dependency-cruiser's and steiger's own versions.
