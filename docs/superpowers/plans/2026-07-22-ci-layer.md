# CI Layer Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** GitHub Actions CI with a single aggregate-success gate, branch protection as code, and Renovate config for `alpcanaydin/recompose`.

**Architecture:** One `ci.yml` runs all jobs in parallel; a final `ci-success` job `needs` them all and is the only required status check. Branch protection lives as a ruleset JSON applied idempotently via `gh api`. Renovate runs as the hosted GitHub App reading `renovate.json`.

**Tech Stack:** GitHub Actions, dorny/paths-filter, gitleaks-action, zizmor, actionlint, pnpm audit, commitlint, GitHub rulesets API, Renovate.

## Global Constraints

- Spec: `docs/superpowers/specs/2026-07-22-ci-layer-design.md`.
- All third-party actions pinned to a commit SHA with a `# vX` comment after it (Renovate updates pins by reading that comment — this is the sole permitted comment form).
- `permissions: contents: read` at workflow level; escalate per job only where needed.
- Runner: `ubuntu-latest` only.
- The repository owner's private alias must never appear in any file, commit message, branch name, or PR. Machine checks cover file contents (`forbidden-owner-alias` rule in `.gitleaks.toml`, pre-commit and CI); commit messages, branch names, and PR titles/bodies rely on review (ADR-0011).
- Every commit message follows caveman-commit (Conventional Commits, terse, no AI attribution).
- This is CI configuration: the verification battery in Task 5 is the test layer. No unit tests apply.

### Task 1: `.github/workflows/ci.yml`

**Files:**

- Create: `.github/workflows/ci.yml`

**Interfaces:**

- Produces: status check named `ci-success` (Task 2's ruleset requires exactly this string), job conditions driven by `changes.outputs.code` / `changes.outputs.workflows`.

- [ ] **Step 1: Resolve action SHAs**

Run each command; note the returned 40-char SHA:

```sh
gh api repos/actions/checkout/commits/v5 --jq .sha
gh api repos/pnpm/action-setup/commits/v4 --jq .sha
gh api repos/actions/setup-node/commits/v5 --jq .sha
gh api repos/dorny/paths-filter/commits/v3 --jq .sha
gh api repos/gitleaks/gitleaks-action/commits/v2 --jq .sha
```

- [ ] **Step 2: Write the workflow**

Create `.github/workflows/ci.yml` with the content below, replacing each `<SHA-...>` with the SHA from Step 1:

```yaml
name: ci

on:
  pull_request:
    types: [opened, synchronize, reopened, edited]
  push:
    branches: [main]

permissions:
  contents: read

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: ${{ github.ref != 'refs/heads/main' }}

jobs:
  changes:
    runs-on: ubuntu-latest
    outputs:
      code: ${{ steps.filter.outputs.code }}
      workflows: ${{ steps.filter.outputs.workflows }}
    steps:
      - uses: actions/checkout@<SHA-checkout> # v5
      - id: filter
        uses: dorny/paths-filter@<SHA-paths-filter> # v3
        with:
          predicate-quantifier: every
          filters: |
            code:
              - '!docs/**'
              - '!**/*.md'
            workflows:
              - '.github/workflows/**'

  check:
    needs: changes
    if: needs.changes.outputs.code == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<SHA-checkout> # v5
      - uses: pnpm/action-setup@<SHA-pnpm> # v4
      - uses: actions/setup-node@<SHA-setup-node> # v5
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm run fmt:check
      - run: pnpm exec turbo run lint typecheck build test

  gitleaks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<SHA-checkout> # v5
        with:
          fetch-depth: 0
      - uses: gitleaks/gitleaks-action@<SHA-gitleaks> # v2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}

  audit:
    needs: changes
    if: needs.changes.outputs.code == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<SHA-checkout> # v5
      - uses: pnpm/action-setup@<SHA-pnpm> # v4
      - uses: actions/setup-node@<SHA-setup-node> # v5
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm audit --prod --audit-level=high

  zizmor:
    needs: changes
    if: needs.changes.outputs.workflows == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<SHA-checkout> # v5
      - run: pipx run zizmor --min-severity medium .github/workflows/

  actionlint:
    needs: changes
    if: needs.changes.outputs.workflows == 'true'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<SHA-checkout> # v5
      - run: go run github.com/rhysd/actionlint/cmd/actionlint@v1.7.7 -color

  commitlint-pr:
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@<SHA-checkout> # v5
      - uses: pnpm/action-setup@<SHA-pnpm> # v4
      - uses: actions/setup-node@<SHA-setup-node> # v5
        with:
          node-version: 22
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - env:
          PR_TITLE: ${{ github.event.pull_request.title }}
        run: printf '%s\n' "$PR_TITLE" | pnpm exec commitlint

  ci-success:
    needs: [changes, check, gitleaks, audit, zizmor, actionlint, commitlint-pr]
    if: always()
    runs-on: ubuntu-latest
    steps:
      - if: contains(needs.*.result, 'failure') || contains(needs.*.result, 'cancelled')
        run: exit 1
```

Notes locked into this design:

- `pull_request.types` includes `edited` because a squash-merge takes the PR title as commit subject; a title fixed or broken after open must re-run `commitlint-pr`.
- `PR_TITLE` goes through `env`, never inline in `run` — template injection guard (zizmor flags the inline form).
- `ci-success` passes when a needed job is `skipped` (docs-only PRs) and fails on `failure`/`cancelled`.

- [ ] **Step 3: Lint the workflow locally**

```sh
go run github.com/rhysd/actionlint/cmd/actionlint@v1.7.7 -color
command -v pipx >/dev/null || brew install pipx
pipx run zizmor --min-severity medium .github/workflows/ci.yml
```

Expected: both exit 0. Fix any finding before proceeding (unpinned action, injection, missing permissions).

- [ ] **Step 4: Commit and push, watch the first run**

```sh

git commit -m "ci: add aggregate-success workflow"
git push

```

Expected: push-triggered run on `main` goes green (`commitlint-pr` skipped — not a PR; `zizmor`/`actionlint` run — workflows changed).

### Task 2: Branch protection ruleset

**Files:**

- Create: `.github/rulesets/main.json`
- Create: `scripts/apply-ruleset.sh`

**Interfaces:**

- Consumes: status check name `ci-success` from Task 1.

- [ ] **Step 1: Write the ruleset JSON**

Create `.github/rulesets/main.json`:

```json
{
  "name": "main",
  "target": "branch",
  "enforcement": "active",
  "conditions": {
    "ref_name": {
      "include": ["~DEFAULT_BRANCH"],
      "exclude": []
    }
  },
  "bypass_actors": [
    {
      "actor_id": 5,
      "actor_type": "RepositoryRole",
      "bypass_mode": "always"
    }
  ],
  "rules": [
    { "type": "deletion" },
    { "type": "non_fast_forward" },
    { "type": "required_linear_history" },
    {
      "type": "pull_request",
      "parameters": {
        "required_approving_review_count": 0,
        "dismiss_stale_reviews_on_push": false,
        "require_code_owner_review": false,
        "require_last_push_approval": false,
        "required_review_thread_resolution": false
      }
    },
    {
      "type": "required_status_checks",
      "parameters": {
        "strict_required_status_checks_policy": false,
        "required_status_checks": [{ "context": "ci-success" }]
      }
    }
  ]
}
```

`actor_id: 5` is the built-in Repository admin role — the emergency bypass from the spec.

- [ ] **Step 2: Write the apply script**

Create `scripts/apply-ruleset.sh`:

```sh
#!/bin/sh
set -eu
repo=$(gh repo view --json nameWithOwner -q .nameWithOwner)
ruleset_id=$(gh api "repos/$repo/rulesets" --jq '[.[] | select(.name == "main")][0].id // empty')
if [ -n "$ruleset_id" ]; then
  gh api -X PUT "repos/$repo/rulesets/$ruleset_id" --input .github/rulesets/main.json
else
  gh api -X POST "repos/$repo/rulesets" --input .github/rulesets/main.json
fi
```

```sh
chmod +x scripts/apply-ruleset.sh
```

- [ ] **Step 3: Apply and verify idempotency**

```sh
./scripts/apply-ruleset.sh
./scripts/apply-ruleset.sh
gh api "repos/alpcanaydin/recompose/rulesets" --jq 'map(.name)'
```

Expected: second run updates (no duplicate); final output `["main"]`.

- [ ] **Step 4: Verify direct push is rejected**

Ruleset bypass for admins is `"always"`, so a plain push may pass for the owner; verify the ruleset itself instead:

```sh
gh api "repos/alpcanaydin/recompose/rulesets" --jq '.[0].id' | xargs -I{} gh api "repos/alpcanaydin/recompose/rulesets/{}" --jq '{enforcement, rules: [.rules[].type]}'
```

Expected: `enforcement: "active"` and rules `["deletion", "non_fast_forward", "required_linear_history", "pull_request", "required_status_checks"]`.

- [ ] **Step 5: Commit**

From here on `main` requires a PR, so create the working branch first, then commit:

```sh
git checkout -b ci-layer
git add .github/rulesets/main.json scripts/apply-ruleset.sh
git commit -m "ci: add branch protection ruleset as code"
```

(Remaining tasks commit onto `ci-layer`; the branch merges via the verification PR in Task 5.)

### Task 4: ADR-0007

**Files:**

- Create: `docs/adr/0007-ci-layer.md`
- Modify: `docs/adr/README.md` (index table)

- [ ] **Step 1: Write the ADR**

Create `docs/adr/0007-ci-layer.md` following the lightweight format of `docs/adr/0006-local-quality-gate-layer.md`:

- **Context**: remote now exists; phase 2 of the gate work; spec reference.
- **Decision**: single `ci.yml` + `ci-success` aggregate (ruleset never changes when jobs are added); ruleset JSON + `gh api` script over Terraform (no state, no toolchain) and over manual UI (versionless); hosted Renovate app over self-hosted (no token upkeep) and over Dependabot (no `minimumReleaseAge`); SHA-pinned actions.
- **Alternatives**: per-concern workflows (required-check list maintained by hand), merge queue (single maintainer, no volume), Dependabot, Terraform.
- **Consequences**: good — one required check forever, docs-only PRs stay cheap via paths-filter, supply-chain cooldown on every update; bad — admin bypass weakens protection for the owner account, `pnpm audit` has no local counterpart so failures surface only in CI, hosted Renovate sees repo metadata.
